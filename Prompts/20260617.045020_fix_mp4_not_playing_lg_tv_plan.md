# User Prompt

The app starts fine and the dlna server is discoverable by my 13 year old LG smart TV LG (LG 47LA6200).
But on clicking the MP4, it does not play. It gives an error 'file type not supported'.
This same MP4 is playing when played from a USB drive

---

# Fix: MP4 "File Type Not Supported" on LG 47LA6200 DLNA

## Background & Root Cause Analysis

The LG 47LA6200 is a 2013 NetCast-era TV. Its built-in DLNA client is notoriously strict about the `DLNA.ORG_PN` (Profile Name) sent in `protocolInfo`. There are **three compounding issues** found in the current code that together cause the "file type not supported" error:

---

## Issue 1 — Wrong / Too-Restrictive DLNA Profile Name for MP4

**File:** `MimeTypeResolver.java` (line 37)

```java
// CURRENT (wrong for general MP4s):
MIME_TO_DLNA.put("video/mp4", "AVC_MP4_MP_SD_AAC_MULT5");
```

The profile `AVC_MP4_MP_SD_AAC_MULT5` maps to the DLNA specification for **SD-only, Main Profile H.264, multi-channel AAC** — a very narrow profile. If the TV checks the PN and the actual file is HD (720p/1080p), or uses a different AAC subtype, the TV will reject it outright with "file type not supported".

The correct fix is to use **multiple fallback profiles** OR use the most broadly-compatible known profile. Community research (MiniDLNA, Emby LG profiles) confirms:
- `AVC_MP4_BL_L3L_SD_AAC` — works for Baseline SD
- `AVC_MP4_MP_SD_AAC_MULT5` — works for Main SD
- `AVC_MP4_HP_HD_AAC` — works for High Profile HD

**Best approach for this old TV**: Advertise **multiple profiles** in the `protocolInfo` by emitting one `<res>` element per known profile, OR use the **broadest compatible set**. The most proven fix is to advertise `AVC_MP4_MP_HD_720p_AAC` (for 720p) and `AVC_MP4_MP_HD_1080i_AAC` (for 1080), plus `AVC_MP4_BL_L3L_SD_AAC` for SD. However, the simplest and most tested approach for LG NetCast is to **omit the PN entirely** for MP4 and rely only on MIME + FLAGS, OR use `AVC_MP4_MP_SD_MPEG1_L3` as a broader fallback.

> The cleanest proven fix: advertise the same file under **multiple `<res>` elements** with different DLNA.ORG_PN values. The TV picks the one it recognises. This is what MiniDLNA and Emby do for LG devices.

---

## Issue 2 — Missing `realTimeInfo.dlna.org` HTTP Response Header

**File:** `HttpStreamingServer.java` (lines 166-174)

The LG 47LA6200 NetCast client requires `realTimeInfo.dlna.org: DLNA.ORG_TLAG=*` on HTTP responses for streaming video. Without it, some LG NetCast firmware versions will stall or report "unsupported" after the initial handshake.

```java
// MISSING (must be added):
exchange.getResponseHeaders().set("realTimeInfo.dlna.org", "DLNA.ORG_TLAG=*");
```

---

## Issue 3 — Single `<res>` Element Limits TV Decoder Selection

**File:** `ContentDirectoryService.java` (buildItem() method)

Currently each MP4 item is advertised with exactly one `<res>` element. The LG 47LA6200 expects the server to enumerate multiple `<res>` elements for the same item, each with a different `DLNA.ORG_PN`, so it can match its hardware decoder. This is the same technique used by MiniDLNA.

The fix: for `video/mp4`, add **three `<res>` elements** with the following profiles (in order of preference):
1. `AVC_MP4_MP_HD_1080i_AAC` — Full HD, Main Profile
2. `AVC_MP4_MP_HD_720p_AAC` — 720p, Main Profile  
3. `AVC_MP4_BL_L3L_SD_AAC` — SD, Baseline Profile (broadest compat)

This mirrors MiniDLNA behaviour for LG NetCast TVs.

---

## Proposed Changes

### Component 1: MimeTypeResolver

#### [MODIFY] MimeTypeResolver.java

- Replace the single MP4 profile with a new constant list `VIDEO_MP4_PROFILES` = the three profiles above.
- Add a new public static method `getVideoMp4Profiles()` returning the ordered list.
- Keep `getDlnaProfile()` intact for non-MP4 types (audio, image, etc.).

---

### Component 2: ContentDirectoryService

#### [MODIFY] ContentDirectoryService.java

- In `buildItem()`, when the item is `video/mp4`, loop over `MimeTypeResolver.getVideoMp4Profiles()` and add **one `Res` object per profile** to the `VideoItem`.
- Keep the existing single-res logic for all other MIME types.

---

### Component 3: HttpStreamingServer

#### [MODIFY] HttpStreamingServer.java

- Add `realTimeInfo.dlna.org: DLNA.ORG_TLAG=*` header to all HTTP responses for audio/video (not images).
- For the `contentFeatures.dlna.org` header on MP4 responses, use the **first (highest priority)** profile `AVC_MP4_MP_HD_1080i_AAC`.

## Open Questions / Notes

> If after this fix the MP4 still won't play, the codec **inside** your specific MP4 is the issue (e.g., H.265/HEVC, or DTS audio). The LG 47LA6200 only supports H.264 (up to Level 4.2) and AAC/AC3 audio via DLNA. Re-encoding with HandBrake to H.264+AAC would be needed in that case.
